アコーディオン

デザイントークンは、色・スペース・タイポグラフィなどの視覚的属性を名前付きで管理する仕組みです。変数として定義することで、テーマ変更やダークモード対応が容易になります。
インタラクティブな要素のタッチ/クリック領域は最低44×44pxを確保するという原則です。Apple HIGとWCAG AAA基準に基づいています。
CSS変数で色をセマンティックに定義し、@media (prefers-color-scheme: dark)と[data-theme="dark"]で変数を上書きします。JSではDS.Theme.toggle()を呼ぶだけです。

黄金律 — 8 Golden Rules

Material 3 / Apple HIG / Polaris / Primer / Linear / Notion の共通原則から抽出

1
8px Grid Law — 全スペースは8の倍数
採用理由
6DS中5つが採用。ディスプレイのサブピクセルと相性が良く、4pxとも整合する最小公倍数。
捨てた代替案
5pxグリッド（割り切れない）、10px（細粒度不足）、Tailwind4px（選択肢過多）
UX効果
視覚的リズム統一、デザイナー↔エンジニア認識ズレ解消、余白の一貫性
2
3-Tier Token — Reference → Semantic → Component
採用理由
Material 3が体系化、Primer・Polarisも採用。ブランド変更・テーマ切替・コンポーネント調整が独立して可能。
捨てた代替案
フラットトークン（変更時全ファイル修正）、2層（semantic省略でボタン/リンクが別定義）
UX効果
テーマ切替コスト激減、開発者がデザイン意図を読みやすい、A/Bテスト容易
3
44px Touch Law — タッチ領域は44×44px以上
採用理由
Apple HIG 44pt / Material 48dp / WCAG AAA 44px。誤タップ率が44px未満で3倍増加（Maryland大学研究）。
捨てた代替案
32px（デスクトップのみなら許容だがモバイル不可）、視覚サイズ=タッチ領域（当たり判定拡張で解決すべき）
UX効果
高齢者・運動障害ユーザーの精度向上、rage tap削減、片手操作性向上
4
Semantic Color — 色はrole名で参照
採用理由
全6DSが実装。ダーク/ライト・ブランド変更・アクセシビリティ調整がコード変更なしで実現。
捨てた代替案
直接hex参照（ブランド変更時に全ファイル検索置換）、色名参照（意図が消える）、Tailwindクラス直打ち（HTML全体の修正になる）
UX効果
ダークモード対応がCSS変数差し替えだけで完結、コードレビュー高速化
5
Relative Type — フォントサイズはremベース
採用理由
PrimerとApple HIGが最も厳格に適用。px固定はブラウザのフォントサイズ設定を無視しWCAG 1.4.4違反リスク。
捨てた代替案
px固定（ユーザー設定を上書き）、em（ネスト計算複雑）、vw（可読性保証困難）
UX効果
OS・ブラウザのフォントサイズ設定が反映、ズームしても比率が崩れない
6
Dark-First Token — CSS変数でダーク/ライト両対応
採用理由
LinearとApple HIGがダークモードに最も注力。ユーザーの約50%がダークモード使用（2024年）。
捨てた代替案
別CSSファイル（同期漏れ）、JSで全カラー書き換え（FOUC発生）、ライトのみ（ユーザー設定無視）
UX効果
目の疲労軽減、OLED消費電力削減、OS設定を尊重した体験
7
WCAG AA Contrast — テキスト4.5:1 / UI 3:1
採用理由
全6DSがアクセシビリティを明記。法的要件（ADA、EAA）にも関係。高コントラストは健常者にも読みやすい。
捨てた代替案
AAAレベル7:1（ビジュアル表現幅が大幅制限）、無チェック（法的リスク+ユーザー離脱）
UX効果
低視力・色覚多様性への対応、屋外視認性向上、高齢者の読みやすさ改善
8
Motion Respect — prefers-reduced-motionに対応
採用理由
Apple HIG・Material 3が明示。前庭障害ユーザーにとってアニメーションが身体的苦痛を引き起こすケースあり。
捨てた代替案
全アニメーション廃止（UX単調化）、設定内OFFのみ（発見されにくい）
UX効果
前庭障害ユーザーの苦痛防止、集中ユーザーへの配慮、バッテリー消費低減
